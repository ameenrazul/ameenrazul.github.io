Volatility

Volatility is a free memory forensics tool developed and maintained by Volatility Foundation, commonly used by malware and SOC analysts within a blue team or as part of their detection and monitoring solutions. Volatility is written in Python and is made up of python plugins and modules designed as a plug-and-play way of analyzing memory dumps.

Volatility is available for Windows, Linux, and Mac OS and is written purely in Python.

✅ Summary of Volatility Overview:

  • Volatility is the most widely used tool for extracting digital artifacts from RAM (volatile memory).
  • It works independently of the system under investigation, but provides deep visibility into its runtime state.
  • Volatility uses a plugin-based architecture to analyze memory dumps.
  • First step in analysis: Identify the image type of the memory dump.
  • Once identified, you can use various Volatility plugins for analysis (to be covered later).
  • There are two versions:
    • Volatility 2 (Python 2-based, older)
    • Volatility 3 (Python 3-based, current and recommended)
      • GitHub: https://github.com/volatilityfoundation/volatility3
  • ⚠️ Volatility 3 syntax is different from Volatility 2; use the latest syntax in your analysis.

In short: Volatility helps extract and investigate data from memory dumps using plugins, and Volatility 3 (Python 3) is the version you should use going forward.

Extracting a memory dump can be performed in numerous ways, varying
based on the requirements of your investigation. Listed below are a few
of the techniques and tools that can be used to extract a memory from a
bare-metal machine.

  • FTK Imager
  • Redline
  • DumpIt.exe
  • win32dd.exe / win64dd.exe
  • Memoryze
  • FastDump

When using an extraction tool on a bare-metal host, it can usually
take a considerable amount of time; take this into consideration during
your investigation if time is a constraint.

Most of the tools mentioned above for memory extraction will output a
.raw file with some exceptions like Redline that can use its own agent
and session structure.

For virtual machines, gathering a memory file can easily be done by
collecting the virtual memory file from the host machine’s drive. This
file can change depending on the hypervisor used; listed below are a few
of the hypervisor virtual memory files you may encounter.

  • VMWare - .vmem
  • Hyper-V - .bin
  • Parallels - .mem
  • VirtualBox - .sav file *this is only a partial memory file

By default, Volatility comes with all existing Windows profiles from Windows XP to Windows 10.

Image profiles can be hard to determine if you don't know exactly what version and build the machine you extracted a memory dump from was. In some cases, you may be given a memory file with no other context, and it is up to you to figure out where to go from there. In that case,
Volatility has your back and comes with the imageinfo plugin. This plugin will take the provided memory dump and assign it a list of the best possible OS profiles. OS profiles have since been deprecated with Volatility3, so we will only need to worry about identifying the profile if using Volatility2; this makes life much easier for analyzing memory dumps.

Note: imageinfo is not always correct and can have varied results depending on the provided dump; use with caution and test multiple profiles from the provided list.

If we are still looking to get information about what the host is running from the memory dump, we can use the following three plugins windows.info linux.info mac.info. This plugin will provide information about the host from the memory dump.

Syntax: python3 vol.py -f <file> windows.info

📌 Syntax Cheat Sheet

python3 vol.py -f <memory_file> windows.pslist       # Basic process listing
python3 vol.py -f <memory_file> windows.psscan       # Hidden process detection
python3 vol.py -f <memory_file> windows.pstree       # Process tree hierarchy
python3 vol.py -f <memory_file> windows.netstat      # Network connections
python3 vol.py -f <memory_file> windows.dlllist      # DLLs loaded per process

🕵️‍♂️ Volatility Malware Hunting Essentials

🔍 1. malfind – Detect Code Injection in Memory

  • Purpose: Finds injected code in process memory.
  • Detection Method:
    • Scans memory regions that are:
      • Executable (e.g., RWE or RX permissions)
      • Not backed by a file (fileless malware)
  • Output:
    • Process name & PID
    • Offset address
    • Memory region’s:
      • Hex dump
      • ASCII view
      • Disassembly (code instructions)
  • Indicators:
    • Presence of MZ header → possibly PE file
    • Shellcode patterns → needs deeper analysis
  • Syntax:
    python3 vol.py -f <file> windows.malfind

🧬 2. yarascan – Search Memory Using YARA Rules

  • Purpose: Scan memory for known malicious patterns or behaviors.
  • How: Uses YARA rules (either from a .yar file or inline) to look for:
    • Suspicious strings
    • Malware signatures
    • Behavioral indicators
  • Syntax:
    python3 vol.py -f <file> windows.yarascan --yara-rules path/to/rules.yar

🧠 Prerequisite Knowledge

Before diving deep into these plugins, it's important to understand:

  • How code injection works (DLL injection, shellcode, PE injection)
  • Evasion techniques (e.g., unlinking processes, reflective loading)
  • What “normal” memory behavior looks like, so you can detect anomalies

🧠 Advanced Memory Forensics in Volatility

When facing stealthy or advanced malware (especially rootkits), traditional process/network/dll listing isn't always enough. These threats may hook into system internals, hide drivers, or manipulate kernel structures to evade detection.

🔗 1. SSDT Hook Detection – ssdt Plugin

  • What is SSDT?

    The System Service Descriptor Table is used by the Windows kernel to resolve system calls.

  • Hooking Technique:

    Malware modifies SSDT entries to redirect kernel functions to malicious code.

  • Use this to detect:

    Inline hooks or function pointer redirection in system services.

  • Note: Legit programs may also hook SSDT; not all results are malicious.
  • Best used when: You already have indicators of compromise.
  • Command:
    python3 vol.py -f <file> windows.ssdt

🚚 2. Kernel Module & Driver Detection

📄 modules Plugin

  • Purpose: Lists all loaded kernel modules (e.g., drivers like .sys files).
  • Use case: Spot malicious drivers pretending to be legitimate ones.
  • Limitation: If a driver is hidden or unloaded, it may not show up.
  • Best used when: You have a filename or signature to look for.
  • Command:
    python3 vol.py -f <file> windows.modules

🔍 driverscan Plugin

  • Purpose: Actively scans for driver objects in memory.
  • Use case: Useful if modules misses something due to hiding/unlinking.
  • Limitation: Might produce false positives or no results if kernel is clean.
  • Command:
    python3 vol.py -f <file> windows.driverscan

🔬 Other Advanced Plugins You Might Explore

🧪 Investigation Workflow (With .raw or .vmem files)

Assuming you’re analyzing Investigation-2.raw:

  1. Get System Info:
    python3 vol.py -f Investigation-2.raw windows.info
  1. List and Analyze Processes:
    python3 vol.py -f Investigation-2.raw windows.pslist
python3 vol.py -f Investigation-2.raw windows.pstree
python3 vol.py -f Investigation-2.raw windows.psscan
  1. Check for Code Injection:
    python3 vol.py -f Investigation-2.raw windows.malfind
  1. Dump DLLs & Network:
    python3 vol.py -f Investigation-2.raw windows.dlllist
python3 vol.py -f Investigation-2.raw windows.netstat
  1. Advanced Malware Hunting:
    python3 vol.py -f Investigation-2.raw windows.ssdt
python3 vol.py -f Investigation-2.raw windows.modules
python3 vol.py -f Investigation-2.raw windows.driverscan
python3 vol.py -f Investigation-2.raw windows.yarascan --yara-rules myrules.yar

✅ Final Tip:

Start with basic process and injection checks, then move to advanced plugins if you suspect rootkit-level tampering or stealthy malware.

  1. What user-agent was employed by the adversary in Case 001?
    vol -f Investigation-1.vmem -o . windows.memmap.Memmap --pid 1640 --dump

strings pid.1640.dmp| grep -i "user-agent"

What you did:

  • You ran Volatility’s process memory dumping plugin (windows.memmap.Memmap with -dump) on a specific process (by PID) inside your memory dump file.
  • This extracted the entire memory content of that running process from the RAM image and saved it as a .dmp file on disk.
  • Then, you ran the strings command on that .dmp file to extract all human-readable text inside that process’s memory.
  • You filtered (grep -i "user-agent") to find any lines containing the text “User-Agent” — a common HTTP header that identifies the client/browser to servers.

Why did we do this?

  • The User-Agent string is usually stored in memory when a program makes HTTP requests.
  • Malware or an attacker’s tool that communicates over HTTP will often store their User-Agent string in their process memory.
  • By dumping the memory of a suspicious process and scanning for User-Agent strings, you can find exactly what User-Agent the adversary used.
