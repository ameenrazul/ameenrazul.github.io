Snort is an open-source, rule-based Network Intrusion Detection and Prevention System (NIDS/NIPS) used to detect and optionally prevent malicious network activity.

üß† Snort Capabilities

  • üü¢ Live Traffic Analysis ‚Äì Real-time monitoring of packet data
  • üîç Attack & Probe Detection ‚Äì Identifies known attack signatures (e.g., port scans, buffer overflows)
  • üìú Packet Logging ‚Äì Records traffic data for analysis and investigation
  • üì∂ Protocol Analysis ‚Äì Parses and understands protocols (TCP, UDP, ICMP, etc.)
  • üö® Real-time Alerting ‚Äì Alerts based on rule matches (e.g., via syslog or console)
  • üß© Modular & Extensible ‚Äì Supports plugins and pre-processors for added detection logic
  • üíª Cross-Platform ‚Äì Works on Linux & Windows

üîß Snort Modes of Operation

ModeDescription
üß™ Sniffer ModeReads and displays packet data like tcpdump
üìÅ Packet Logger ModeLogs packets to disk for later analysis
üõ°Ô∏è NIDS/NIPS ModeDetects and/or prevents malicious packets based on rules
  • NIDS (Detection): Alerts on malicious traffic but does not block it
  • NIPS (Prevention): Actively blocks malicious packets (when deployed inline)

üìÅ Snort Configuration File (snort.conf)

The Snort configuration file is the central brain of a Snort deployment. It defines how Snort behaves during runtime.

‚úÖ Key Functions of the Configuration File:

  • üìú Rules: Which detection rules to load
  • üß© Plugins & Preprocessors: Enable modules for deeper inspection (e.g., frag3, stream5)
  • ‚öôÔ∏è Detection Settings: Define what constitutes suspicious traffic
  • üì§ Output Options: How alerts are logged (console, file, unified2, etc.)
  • üîÅ Default Actions: Whether to alert, log, or drop packets
üóÇÔ∏è You can create multiple configuration files for different use cases (e.g., internal vs external traffic monitoring), but only one can be used at a time during execution.

Great! This task was a hands-on demonstration of how to investigate PCAP files using Snort, using various command-line parameters to process one or more PCAPs effectively. Let‚Äôs review and break it down into key learnings:

üîç Snort‚Äôs PCAP Analysis Modes Recap

ParameterPurpose
-r <pcap>Read a single PCAP file.
--pcap-list="<file1> <file2>"Process multiple PCAPs.
--pcap-showShow PCAP name when processing ‚Äì useful for tracing alerts per file.

‚úÖ Example Commands & Their Use

1. Single PCAP Investigation

sudo snort -c /etc/snort/snort.conf -q -r icmp-test.pcap -A console -n 10
  • c: Specifies the config file.
  • q: Quiet mode (suppress banner and runtime messages).
  • r: Input PCAP file.
  • A console: Output alerts to the console.
  • n 10: Process only the first 10 packets.

üìå Result: Alerts triggered by ICMP packets in icmp-test.pcap based on the rule [1:10000001:0] ICMP Packet found.

2. Multiple PCAPs with -pcap-list

sudo snort -c /etc/snort/snort.conf -q --pcap-list="icmp-test.pcap http2.pcap" -A console
  • This runs Snort on both PCAPs, but doesn‚Äôt show which alert came from which file.

üìå Limitation: You can‚Äôt easily trace alerts back to the source PCAP.

3. Multiple PCAPs with -pcap-show

sudo snort -c /etc/snort/snort.conf -q --pcap-list="icmp-test.pcap http2.pcap" -A console --pcap-show
  • Adds PCAP filename before processing.
  • Extremely useful in investigations where alerts need to be correlated with the original PCAP.

üìå Result: Each alert is shown under the corresponding file name:

Reading network traffic from "icmp-test.pcap"
[alert logs]
Reading network traffic from "http2.pcap"
[alert logs]

üß† Key Takeaways

  • Snort can analyze past captures (pcap files), not just live traffic.
  • The ruleset in snort.conf determines what alerts are generated.
  • Use -pcap-show with -pcap-list to separate and identify which alert came from which file ‚Äî a must for investigations!
  • The ICMP rule you wrote (likely in local.rules) is correctly detecting echo requests/replies, confirming proper rule syntax and Snort configuration.
